# 心愿便利贴 PRD v0.1

## 1. 背景

当前赠品采购预测工作台主要服务运营侧的采购测算、消耗查看和活动复盘。为了更早收集一线销售对商品的真实倾向，需要新增一个轻量互动模块，让销售每周参与投票，帮助运营判断哪些品更可能成为下月爆品。

新模块命名为「心愿便利贴」。整体体验不做传统后台风格，而是偏轻松、有趣的活动页，让销售像贴便利贴一样表达“我觉得客户会喜欢什么品”。

## 2. 目标

- 收集销售对候选品的偏好。
- 每名销售每周最多 3 次投票机会。
- 支持销售给商品补充标签或推荐理由。
- 运营可查看排行榜、Top 8、总票数、参与人数、部门/团队分布、品类分布、标签热词。
- 运营基于投票结果选出下个月爆品候选。

## 3. 用户角色

### 3.1 销售用户

- 登录后参与投票。
- 查看本周剩余票数。
- 浏览运营指定的候选品。
- 给候选品投票。
- 给商品新增标签或推荐理由。
- 投票后查看本周排行榜。

### 3.2 运营用户

- 配置本周候选品。
- 配置投票周期和每人投票次数。
- 查看投票结果。
- 查看 Top 8 爆品候选。
- 查看总票数、参与人数、部门/团队分布、品类分布、标签热词。
- 按周查看历史结果。

## 4. 核心流程

1. 运营创建投票周期。
2. 运营配置本周候选品。
3. 销售进入「心愿便利贴」。
4. 系统识别销售身份。
5. 销售浏览候选品便利贴卡片。
6. 销售投票，并可补充标签或推荐理由。
7. 投票成功后展示反馈和排行榜。
8. 每周结束后，运营查看结果。
9. 运营取前 8 名作为下月爆品候选。

## 5. MVP 范围

第一版先跑通完整闭环：

- 候选品管理。
- 销售投票。
- 每人每周 3 票限制。
- 同一周期内默认不允许同一销售重复投同一个商品。
- 标签/理由补充。
- 排行榜。
- 运营结果页。
- Top 8 展示。
- 背景音乐播放/暂停。

第一版暂不做：

- 多首音乐切换。
- 复杂动效配置后台。
- 自动生成采购建议。
- 与现有采购活动强绑定。
- 复杂权限系统。

## 6. 页面规划

### 6.1 销售投票页 `/wish-notes`

页面元素：

- 模块标题：心愿便利贴。
- 右上角音乐播放/暂停按钮。
- 当前用户信息。
- 本周剩余票数。
- 候选品便利贴卡片列表。
- 投票按钮：贴上我的心愿。
- 标签/理由输入入口。
- 投票成功反馈。
- 排行榜入口。

交互要求：

- 商品卡片尽量做成便利贴/心愿墙视觉。
- 投票动作要有轻量反馈，例如卡片上墙、按钮变化或成功提示。
- 投票后引导用户查看排行榜。

### 6.2 排行榜页 `/wish-notes/ranking`

页面元素：

- 本周心愿榜。
- Top 8 高亮。
- 每个商品票数。
- 标签热词。
- 我的投票记录。

### 6.3 运营管理页 `/wish-notes/admin`

页面元素：

- 投票周期管理。
- 新增候选品。
- 编辑候选品。
- 上架/下架候选品。
- 配置每人每周投票次数，默认 3 次。

### 6.4 运营结果页 `/wish-notes/admin/results`

页面元素：

- 周期筛选。
- Top 8 爆品候选。
- 全量排行榜。
- 总票数。
- 参与人数。
- 部门/团队分布。
- 品类分布。
- 标签热词。
- 导出入口，后续迭代。

## 7. UI 和交互方向

整体不走后台工具风格，偏轻松活动页风格。

关键词：

- 便利贴。
- 心愿墙。
- 排行榜。
- 轻松有趣。
- 有参与感。
- 有反馈感。

建议文案：

- 投票：贴上我的心愿。
- 排行榜：本周心愿榜。
- Top 8：下月爆品候选清单。
- 标签：补一句销售理由。

音乐播放要求：

- 默认不自动播放，避免浏览器限制和打扰用户。
- 用户点击后播放/暂停。
- 一期只支持一首固定背景音乐。
- 音乐只增强氛围，不影响投票主流程。

## 9. API 设计

### 9.1 销售端 API

- `GET /api/wish-notes/period/current`：获取当前投票周期。
- `GET /api/wish-notes/products`：获取候选品列表。
- `GET /api/wish-notes/me`：获取当前用户本周投票情况。
- `POST /api/wish-notes/votes`：提交投票。
- `GET /api/wish-notes/ranking`：获取排行榜。

### 9.2 运营端 API

- `GET /api/wish-notes/admin/periods`：获取投票周期列表。
- `POST /api/wish-notes/admin/periods`：创建投票周期。
- `POST /api/wish-notes/admin/products`：新增候选品。
- `PUT /api/wish-notes/admin/products/:id`：编辑候选品。
- `GET /api/wish-notes/admin/results`：获取运营结果看板。

## 10. 核心规则

### 10.1 投票限制

- 同一销售在同一周期最多投 3 次。
- 同一销售在同一周期默认不能重复投同一个商品。
- 投票周期结束后不可继续投票。
- 下架商品不可继续投票。

### 10.2 排行榜

- 默认按商品总票数倒序。
- Top 8 强高亮展示。
- 票数相同时，可按最近投票时间或运营排序作为次级排序。

### 10.3 标签统计

- 用户提交的标签进入标签热词统计。
- 标签热词用于帮助运营理解销售选择原因。

## 11. 登录和权限

当前方向是后续接 LDAP 登录系统。

第一版开发可以先预留以下用户字段：

- `user_id`
- `user_name`
- `department`
- `team`

MVP 可先用模拟登录、URL 参数或临时用户信息跑通投票闭环。后续接 LDAP 时，替换当前用户信息来源即可，投票和统计逻辑不需要大改。

## 17. 前端交互设计说明 v0.1

### 17.1 整体视觉方向

「心愿便利贴」不是传统后台表单页，而是偏轻互动活动页。销售端要轻松、有参与感；运营端可以保持清晰高效，但也保留模块统一的心愿墙视觉。

视觉关键词：

- 温暖、明亮、轻松。
- 便利贴、贴纸、心愿墙。
- 榜单、徽章、Top 8 高亮。
- 卡片化、圆角、轻阴影。
- 少量动效，不影响操作效率。

建议色彩：

- 主色：暖黄色、橙色或浅粉色，用于便利贴和行动按钮。
- 辅色：浅蓝、浅绿、浅紫，用于不同品类或标签。
- 背景：浅米色或浅渐变，避免纯后台白底。
- 排行榜前三：金、银、铜或更活泼的徽章样式。

### 17.2 销售投票页 `/wish-notes`

页面目标：让销售快速理解玩法、浏览候选品，并完成本周最多 3 次投票。

页面结构：

1. 顶部氛围区
   - 标题：心愿便利贴
   - 副标题：把你觉得客户会喜欢的好物贴上墙
   - 右侧：音乐播放/暂停按钮
   - 入口：查看本周心愿榜

2. 用户和票数区
   - 展示当前用户姓名、部门/团队
   - 展示本周剩余票数，例如：本周还可以贴 2 张心愿
   - 展示投票周期，例如：5.13 - 5.19

3. 候选品心愿墙
   - 使用便利贴卡片展示候选品
   - 卡片可使用轻微错位、旋转、不同底色，增强便利贴感
   - 每张卡片展示商品图、商品名、品类、默认标签、简介
   - 主按钮：贴上我的心愿
   - 次入口：补一句销售理由

4. 投票反馈区
   - 投票成功后显示轻提示，例如：已贴上心愿，去看看它冲到第几名了
   - 提供跳转排行榜按钮

核心交互：

- 点击「贴上我的心愿」前，如果还有票数，弹出标签/理由输入层。
- 标签/理由可选填，但建议引导填写。
- 提交成功后，当前卡片变成已投状态。
- 已投商品按钮文案变为：已贴心愿。
- 剩余票数实时减少。
- 如果票数用完，所有未投商品按钮置灰，文案为：本周心愿已贴满。
- 如果重复投同一商品，提示：这张便利贴你已经贴过啦，换一个品试试。

### 17.3 标签/理由输入层

触发：销售点击候选品卡片上的「贴上我的心愿」。

展示内容：

- 商品名称和缩略图。
- 快捷标签，例如：客户喜欢、转化强、礼盒感强、适合宝妈、适合新人、性价比高。
- 自定义标签输入框。
- 推荐理由输入框，提示文案：为什么你觉得它会爆？
- 确认按钮：确认贴上。
- 取消按钮：再想想。

交互规则：

- 快捷标签可多选。
- 自定义标签长度建议限制在 10 个字以内。
- 推荐理由建议限制在 100 个字以内。
- 标签和理由第一版可选填，不阻断投票。

### 17.4 排行榜页 `/wish-notes/ranking`

页面目标：让销售看到投票后的即时反馈，让运营和销售都能感知哪些品正在成为热门。

页面结构：

1. 顶部区
   - 标题：本周心愿榜
   - 副标题：看看大家都把心愿贴给了谁
   - 返回投票页按钮

2. Top 8 区
   - Top 8 单独高亮展示
   - 前三名使用更明显的榜单徽章
   - 每个商品展示票数、品类、热词标签

3. 全量榜单区
   - 按票数倒序展示所有候选品
   - 展示排名、商品名、票数、热词

4. 我的投票区
   - 展示当前用户本周投过的商品
   - 展示剩余票数

状态说明：

- 无投票数据：展示空状态文案「心愿墙还空着，快来贴下第一张」。
- 当前用户未投票：提示「你还没有贴心愿，本周还有 3 次机会」。
- 周期已结束：展示「本周心愿已收官」。

### 17.5 运营管理页 `/wish-notes/admin`

页面目标：让运营配置投票周期和候选品。

页面风格：可以比销售端更偏管理，但保留卡片和轻活动感。

页面结构：

1. 周期配置区
   - 创建投票周期
   - 设置周期名称、开始时间、结束时间
   - 设置每人可投票次数，默认 3
   - 设置周期状态

2. 候选品管理区
   - 商品列表
   - 新增候选品按钮
   - 编辑、上架、下架操作
   - 按状态筛选：全部、上架、下架

3. 新增/编辑候选品弹层
   - 商品名称
   - 商品图片 URL
   - 品类
   - 成本
   - 默认标签
   - 商品简介
   - 排序
   - 状态

### 17.6 运营结果页 `/wish-notes/admin/results`

页面目标：让运营基于投票数据选出下个月爆品。

页面结构：

1. 筛选区
   - 投票周期选择
   - 部门/团队筛选，第一版可先展示全量，后续再做筛选交互

2. 核心指标区
   - 总票数
   - 参与人数
   - 候选品数量
   - Top 8 数量

3. Top 8 爆品候选区
   - 展示前 8 名商品
   - 展示票数、排名、品类、标签热词
   - 视觉上强调“可用于下月爆品候选”

4. 数据分布区
   - 部门/团队分布
   - 品类分布
   - 标签热词

5. 全量排行榜
   - 商品排名
   - 票数
   - 参与人数
   - 品类
   - 热门标签

### 17.7 音乐播放组件

组件位置建议：`components/wish-notes/MusicPlayer.tsx`。

交互规则：

- 默认不自动播放。
- 初始按钮文案：打开氛围音乐。
- 播放中按钮文案：暂停音乐。
- 用户点击后播放或暂停。
- 记录本次开关状态，刷新页面后尽量保持用户选择。
- 如果浏览器阻止播放，给出轻提示，不影响页面使用。

样式建议：

- 做成右上角小胶囊按钮。
- 可使用音符、唱片或小喇叭图标。
- 播放中可以有轻微旋转或跳动效果。

### 17.8 文案风格

整体文案要轻松、短句、低压力。

推荐文案：

- 心愿便利贴
- 把你觉得客户会喜欢的好物贴上墙
- 本周还可以贴 2 张心愿
- 贴上我的心愿
- 补一句销售理由
- 已贴上心愿
- 本周心愿已贴满
- 去看看本周心愿榜
- 下月爆品候选清单

避免文案：

- 提交投票
- 操作成功
- 数据看板
- 管理后台
- 投票失败

### 17.9 响应式要求

第一版需要保证桌面端可用，小屏幕不崩。

- 桌面端：候选品卡片 3-4 列展示。
- 平板或窄屏：候选品卡片 2 列展示。
- 手机宽度：候选品卡片 1 列展示。
- 运营页表格在小屏可横向滚动。
- 关键按钮在移动端保持可点击，不要过小。

### 17.10 前端状态清单

销售投票页需要覆盖：

- 加载中。
- 无当前投票周期。
- 当前周期未开始。
- 当前周期已结束。
- 无候选品。
- 有候选品但用户未投票。
- 投票成功。
- 票数已用完。
- 重复投票被拦截。
- 接口异常。

排行榜页需要覆盖：

- 加载中。
- 无榜单数据。
- 有榜单数据。
- 周期已结束。

运营页需要覆盖：

- 无投票周期。
- 无候选品。
- 创建成功。
- 保存成功。
- 删除/下架确认。
- 接口异常。
